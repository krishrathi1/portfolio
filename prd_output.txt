1.  Executive Summary
This document defines the complete product requirements for a terminal-based developer portfolio — a web application that emulates a real Unix terminal as its entire user interface. Unlike conventional portfolio sites, every interaction occurs through typed commands, creating an experience that is memorable, technically impressive, and deeply personal to developers.

The portfolio targets two primary audiences: technical recruiters evaluating engineering depth, and fellow developers who appreciate craftsmanship. The terminal metaphor allows the creator to showcase not just what they have built, but how they think.

2.  Product Vision & Goals
2.1  Vision Statement
"To build the most memorable developer portfolio on the internet — one that makes every visitor stop, type, and smile."
2.2  Goals
Achieve 10,000+ unique visitors within 90 days of launch via organic dev community sharing
Maintain average session duration above 4 minutes — 8x the industry average for portfolio sites
Drive 15+ inbound job inquiries per month from senior engineering roles
Earn features in developer newsletters: ByteByteGo, TLDR, CSS Tricks within Q2
Rank on first page of Google for '[your name] developer portfolio'
2.3  Non-Goals
This is NOT a general-purpose terminal emulator — only portfolio-specific commands are supported
No server-side rendering framework required — pure static site deployable on Vercel/Netlify
Not optimised for SEO content crawlers — the JS-heavy nature is intentional and accepted

Layer | Technology | Rationale
Frontend | React 18 + TypeScript | Component model maps cleanly to terminal command registry
Terminal Engine | xterm.js or custom | Full VT100 support; custom parser for pipe/flag syntax
Styling | Tailwind CSS + CSS vars | Theme switching via single root CSS variable swap
Hosting | Vercel (static + edge) | Edge functions for AI + analytics; free tier for static assets
AI Backend | Claude API (Anthropic) | Streaming responses for the ai chat sub-shell command
Data Layer | Vercel KV (Redis) | Rate limiting for AI; visitor analytics persistence
Blog Source | MDX or Notion API | Markdown rendered inline in terminal with syntax highlighting
Integrations | GitHub GraphQL, Spotify | Live data for git log and status --live commands

Risk | Likelihood | Impact | Mitigation
Mobile UX — terminal is keyboard-first | High | High | Command pill bar + touch-friendly autocomplete strip above keyboard
Accessibility — screen readers struggle with terminal | Medium | High | ARIA live regions on output; keyboard-only navigation guaranteed
API rate limits (GitHub, Spotify, Claude) | Medium | Medium | Cache responses in Vercel KV; graceful fallbacks to static data
JS-heavy site hurts SEO | High | Low | Pre-render static meta, OG images; accept trade-off consciously
Claude API costs scale with traffic | Low | Medium | 10 msg/visitor rate limit via KV; edge function enforced
